Scenario 1: Filtering That Works During the Search, Not After It

“battery replacement procedure” — 91% of the corpus is Apple. Without a filter, Sony guides are buried.

Elastic Approach

  • bool.filter with term: brand lives inside the semantic retriever
  • Applied during ANN — recall fully preserved for minority brands
  • Full Query DSL: term, range, bool, nested filters

Competitor Gap

  • Typically limited to simple allow/deny on pre-defined namespaces
  • Filtering is post-ANN — minority brand recall decimated
CapabilityElasticCompetitors
Filter applied during ANN✅ Inside the retriever❌ Post-filter only
Filter expressivenessFull Query DSL (term, range, bool, nested)Simple allow/deny on namespaces
Recall for minority brands✅ Preserved❌ Decimated by majority saturation

One Query. Filter Inside the Retriever.

The brand filter is composed into the semantic retriever — applied during ANN, not as a post-processing step.

retriever={

        "query": {"semantic": {"field": "content", "query": query}},

Scenario 2: A Technician Describes a Symptom. Elastic Understands.

“machine makes grinding noise after startup” — no brand, no model, no part name. Just a symptom.

BM25 (Lexical)

Exact token matching — finds docs with literal “grinding”, “noise”, “machine”

Semantic (Dense Vector)

Understands mechanical failure even without keyword overlap — finds “abnormal fan sound”

Reranker (Jina)

Cross-attention re-reads top-k against query — promotes most diagnostically relevant result to #1

Competitors: Vector search only — no built-in BM25, no hybrid retrieval, no reranker. Requires separate systems.

All Three Strategies. One Platform. One Query.

Scenario 3: A Serial Number Should Return One Generation — Not Five

“battery replacement steps” + serial C02FX123 → should return only 2007 Core 2 Duo guides.

Elastic Approach

  • Range filter on serial_range_start / serial_range_end composed inside the retriever
  • Standard boolean range logic — compound bool.must across multiple fields

Competitor Gap

  • Typically limited to equality and set membership on flat metadata
  • No range queries, no compound boolean logic
  • This query cannot be expressed without a more complex, higher-cost service tier
CapabilityElasticCompetitors
Range filter on metadata✅ range query❌ Equality/set only
Compound boolean across fields✅ bool.must + multiple range❌ Not supported
Serial-scoped search✅ Works natively❌ Requires higher-cost service tier

Scenario 4: A French Technician. English Manuals. Zero Translation.

“l’écran ne répond plus au toucher” — zero English cognates. BM25 returns nothing.

Elastic Approach

  • Jina Embeddings: 119+ languages in a shared vector space — French query retrieves English results
  • Aggregation on language field runs as side-channel on same query — zero extra calls
  • BM25 contrast query demonstrates failure mode, reinforcing why semantic matters

Competitor Gap

  • Cross-lingual similarity is model-dependent — same model works on any platform
  • No lexical search to contrast against
  • Aggregation requires fetching + app-code processing
CapabilityElasticCompetitors
Cross-lingual vector search✅ Model-dependent (Jina v5)✅ Model-dependent (same)
Language facet / aggregation✅ Native aggs❌ Application code required
BM25 failure mode contrast✅ Same notebook cell❌ No lexical search to compare
“Why semantic?” demonstration✅ Side-by-side BM25 vs semantic❌ No contrast available
